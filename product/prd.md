# PRD.md

## 1. 제품 개요

### 제품명
식단메이트

### 제품 한 줄 정의
식단메이트는 사용자가 식단을 **잘 알게 하는 앱**이 아니라, 식단을 **계속 지키게 만드는 실행형 식단 메이트**입니다.

### 제품 목표
사용자가 매일 식단을 완벽하게 통제하도록 만드는 것이 아니라,
아래 3가지를 반복 가능하게 만드는 것이 1차 목표입니다.

1. 오늘 무엇을 먹을지 빠르게 결정한다
2. 식사 내용을 아주 가볍게 기록한다
3. 무너진 뒤에도 다음 끼니부터 복귀한다

---

## 2. 문제 정의

현재 타깃 사용자는 식단 정보를 모르는 것이 아닙니다.
문제는 다음과 같습니다.

### 문제 1. 결정 피로
- 오늘 무엇을 먹을지 매번 새로 결정해야 함
- 건강/다이어트 목표와 현실(외식, 편의점, 배달)이 자주 충돌함

### 문제 2. 기록 피로
- 칼로리 앱은 입력이 귀찮고 오래 못 감
- 숫자를 정확히 입력해야 한다는 부담이 큼

### 문제 3. 복귀 실패
- 한 끼 또는 하루 무너지면 다음날까지 무너짐
- “한 번 실패 = 전체 실패”처럼 느껴짐

### 우리가 해결할 핵심 문제
초기 식단메이트는 아래 한 가지를 가장 먼저 해결해야 합니다.

> **식단을 망치지 않게 하는 앱이 아니라, 망쳐도 다시 이어가게 하는 앱**

---

## 3. 타깃 사용자

## 1차 ICP
20~39세 직장인/프리랜서 남녀 중,
체중 감량 또는 건강한 식습관 개선 의지가 있지만,
식단을 2주 이상 지속하지 못하는 사용자.

### 공통 특징
- 다이어트/건강 의지는 있음
- 정보는 이미 많이 봄
- 현재 pain은 지식 부족이 아니라 실행 지속 실패
- 칼로리 기록형 앱에 피로감을 느낌
- “오늘 뭐 먹지?”가 가장 큰 friction

### 제외할 사용자
초기 MVP에서는 아래 사용자를 핵심 타깃으로 삼지 않습니다.
- 질환 치료/관리 목적 사용자
- 전문 보디빌딩/대회 준비층
- B2B 복지 시장
- 단순 레시피 탐색형 사용자

## 4. JTBD (Jobs To Be Done)

### 메인 JTBD
- 바쁜 하루 중에도 식단 결정을 빠르게 끝내고 싶다
- 식단을 기록하는 부담 없이 흐름만 유지하고 싶다
- 무너진 날에도 죄책감 없이 다시 시작하고 싶다

### 보조 JTBD
- 외식/배달/편의점 같은 현실적인 상황에서도 선택지를 받고 싶다
- 일주일 뒤 돌아봤을 때, 내가 얼마나 유지했는지 간단히 알고 싶다

## 5. 핵심 가치 제안

식단메이트의 초기 핵심 가치는 아래 3개입니다.

1. **빠른 결정**
   - 오늘의 끼니를 빨리 정하게 한다
2. **낮은 입력 마찰**
   - 기록이 귀찮아서 이탈하지 않게 한다
3. **복귀 설계**
   - 실패 후 다시 이어붙이게 만든다

### 포지셔닝 문장
식단메이트는 식단 정보를 많이 주는 앱이 아니라,
**바쁜 사람이 식단을 실제로 계속하게 만드는 초경량 실행 파트너**입니다.

## 6. 사용자 시나리오

## 시나리오 A — 점심 결정
- 사용자는 점심 전에 앱을 연다
- “오늘 점심 뭐 먹지?”를 고민한다
- 앱은 현재 상황(집/밖, 이미 먹은 것, 예산감 등)에 맞는 선택지를 제안한다
- 사용자는 바로 선택한다
- 식단 결정에 걸리는 시간을 줄인다

## 시나리오 B — 식사 후 기록
- 사용자는 식사 후 앱을 연다
- 사진 1장 또는 버튼 1~2회로 기록한다
- 정밀 입력 없이 기록이 남는다

## 시나리오 C — 식단 이탈 후 복귀
- 사용자는 야식/외식/회식 등으로 식단이 무너진다
- 앱은 죄책감 유발 없이 다음 끼니 기준으로 다시 제안한다
- 사용자는 “오늘 망했다”가 아니라 “다음 끼니부터 다시 시작” 경험을 한다

## 시나리오 D — 주간 회고
- 사용자는 주말에 일주일 요약을 본다
- 칼로리 대신
  - 얼마나 자주 기록했는지
  - 얼마나 자주 복귀했는지
  - 얼마나 유지했는지
  를 확인한다

## 7. MVP 범위

## Must-have

### 1) 온보딩
- 목표 선택: 감량 / 유지 / 건강 식습관
- 식단 스타일 선택: 일반식 중심 / 간편식 중심 / 외식 많음
- 제약 선택: 기피 음식 / 예산감 / 요리 가능 여부

### 2) 오늘의 식단 제안
- 아침 / 점심 / 저녁 / 간식 단위 추천
- 외식 / 편의점 / 배달 상황 대응
- 제안은 길지 않고 바로 선택 가능해야 함

### 3) 초저마찰 기록
- 사진 업로드 또는 버튼형 기록
- 기록 선택값 예시:
  - 잘 지켰음
  - 비슷함
  - 벗어남

### 4) 복귀 UX
- 이탈 후 다음 끼니 중심 재설계
- 죄책감보다 복귀를 유도하는 카피

### 5) 주간 피드백
- 기록률
- 유지율
- 복귀율
- 다음 주 한 줄 제안

## 8. MVP 제외 범위

초기에는 아래 기능을 제외합니다.

- 정밀 매크로/칼로리 자동 계산
- 한국 영양 DB 기반 복잡한 영양 최적화 엔진
- 대규모 레시피 콘텐츠 허브
- 커뮤니티/소셜 기능
- 웨어러블/체중계 연동
- 의료/처방형 표현
- 복잡한 결제/플랜 구조
- 매장 단위 최저가 비교 보장

## 9. UX 원칙

### 원칙 1. 입력보다 선택
사용자가 쓰기보다 누르게 해야 합니다.

### 원칙 2. 숫자보다 흐름
초기에는 영양 정밀도보다 행동 지속성이 중요합니다.

### 원칙 3. 죄책감 금지
복귀 경험을 설계해야 합니다.

### 원칙 4. 오늘 우선
초기 제품은 “한 달 전체”보다 “오늘 한 끼”를 더 잘 풀어야 합니다.

## 10. 성공 지표

### 북극성 지표
주간 실행 식사 수 / WAU

정의:
앱이 제안한 끼니 중 사용자가 실제로 선택, 기록, 또는 복귀 행동을 한 식사 수.

### 초기 핵심 KPI
- 첫 세션 식단 생성률
- 첫 기록 전환율
- D7 리텐션
- D30 리텐션
- 7일 내 3회 이상 기록 비율
- 다음 끼니 복귀율
- 추천 후 실행 응답률

### 초기 목표 초안
- 첫 세션 식단 생성률: 65%+
- 첫 가치 도달 시간: 90초 이내
- D7 리텐션: 35%+
- D30 리텐션: 20%+
- 7일 내 3회 이상 기록 비율: 40%+
- 다음 끼니 복귀율: 25%+

## 11. 베타 검증 질문

베타에서 확인해야 할 것은 아래입니다.

1. 사용자가 진짜 가장 힘들어하는 것은 추천 부족인가, 기록 피로인가?
2. 한 끼 추천이 실제 행동으로 이어지는가?
3. 사진/버튼 기반 기록만으로도 충분한가?
4. 복귀 UX가 실제 유지율에 도움을 주는가?
5. 사용자가 구독료를 낼 만큼 반복 가치가 생기는가?

## 12. 리스크

### 리스크 1. 너무 넓은 문제를 풀려 함
→ 초기에는 “식단 지속” 하나에 집중

### 리스크 2. 추천보다 입력 피로가 더 큼
→ 입력 최소화 우선

### 리스크 3. 영양 정밀도를 너무 일찍 약속함
→ 초기에는 생활관리형 가이드로 제한

### 리스크 4. 레시피/커머스/건강관리까지 한 번에 넣으려 함
→ 후순위로 미룸
