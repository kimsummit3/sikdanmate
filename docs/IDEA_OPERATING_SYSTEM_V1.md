# IDEA_OPERATING_SYSTEM_V1.md

## 목적
이 문서는 Jarvis/ai-company/wips/life-style 체계를
단순 채팅 분리가 아니라,
**아이디어 → 회의 → 산출물 → 다음 단계**로 이어지는 운영 시스템으로 고정하기 위한 문서다.

---

## 1. 시스템 목표
사용자는 아이디어를 던지기만 하면 된다.
그 다음은 시스템이 아래를 알아서 처리해야 한다.

1. 아이디어 유형 분류
2. 지금 필요한 회의 단계 판단
3. 담당 agent 지정
4. 산출물 생성
5. 다음 회의 안건 제안

즉, 사용자 경험은 "대화"지만,
내부적으로는 "운영 파이프라인"처럼 돌아가야 한다.

---

## 2. 핵심 원칙
### 원칙 1. 메인은 오케스트레이터
Jarvis(main)는 모든 일을 직접 다 하지 않는다.
주 역할은 아래다.
- 입력 해석
- 우선순위 판단
- 회의 단계 지정
- 적절한 specialist에게 위임
- 결과 회수 및 최종 정리

### 원칙 2. specialist는 책임 단위다
agent는 캐릭터가 아니라 책임 단위다.

- ai-company: 사업/전략/제품 정의/우선순위/시장 판단
- wips: 문서/설계/프로토타입/코드/실행 산출물
- life-style: 생활/루틴/개인 운영/정보성 지원

### 원칙 3. 채널과 agent를 분리한다
채널은 메시지가 들어오는 입구이고,
agent는 일을 처리하는 책임 단위다.
둘을 혼동하지 않는다.

### 원칙 4. 모든 의미 있는 작업은 산출물을 남긴다
- 문서
- 체크리스트
- 화면 구조
- 실행 계획
- 레포 구조
- 코드
- 메모

---

## 3. 아이디어 처리 파이프라인
### 입력 예시
- 이런 서비스 만들면 어떨까?
- 이 기능 붙이면 먹힐까?
- 이런 운영 시스템 필요해
- 이 문제를 어떻게 제품으로 만들지?

### 처리 단계
#### Stage 1. 분류
Jarvis가 먼저 분류한다.
- 제품 아이디어
- 사업/전략 아이디어
- 구현/자동화 요청
- 생활/루틴성 요청
- 리서치 요청

#### Stage 2. 현재 단계 판정
아이디어마다 지금 필요한 회의 단계가 다르다.
가능한 단계는 아래와 같다.
- 정의
- 검증
- 구조 설계
- 디자인
- 구현
- 배포/운영
- 측정/개선

#### Stage 3. 담당 지정
- 정의/전략/시장 = ai-company
- 문서/설계/프로토타입/코드 = wips
- 생활/정보 = life-style
- 전체 조정/종합 = main

#### Stage 4. 회의 실행
각 단계는 회의처럼 진행된다.
회의마다 반드시 아래를 갖는다.
- 이번 회의 목적
- 이번 회의에서 결정할 것
- 이번 회의에서 보류할 것
- 끝나고 남길 산출물

#### Stage 5. 산출물 생성
예:
- PRD
- IA
- 화면 목록
- 실험 계획
- MVP 범위
- 레포 구조
- 티켓

#### Stage 6. 다음 단계 자동 제안
현재 단계가 끝나면 Jarvis는 다음을 제안한다.
예:
- 이제 디자인 안 하면 구현 품질이 떨어집니다
- 이제 가격 구조를 정해야 합니다
- 이제 사용자 인터뷰가 필요합니다

---

## 4. 회의 엔진 구조
### 회의 타입
#### 1) 정의 회의
목적:
- 이 아이디어가 정확히 무엇인지 정의
산출물:
- 한 줄 정의
- 핵심 문제
- 대상 사용자

#### 2) 검증 회의
목적:
- 진짜 문제인지, 돈 낼 가치가 있는지 확인
산출물:
- 검증 질문
- 인터뷰 가이드
- 리스크

#### 3) 구조 설계 회의
목적:
- 제품 구조와 정보 구조 설계
산출물:
- IA
- 시스템 구조
- 모드 분리

#### 4) 디자인 회의
목적:
- UX 흐름/화면 우선순위/상태 설계
산출물:
- 화면 목록
- Figma 기준 설계 문서
- 디자인 원칙

#### 5) 구현 회의
목적:
- 실제 빌드 계획 확정
산출물:
- 구현 순서
- 기술 결정
- 태스크 목록

#### 6) 운영 회의
목적:
- 배포 후 운영/측정/개선 구조 정의
산출물:
- KPI
- 반복 루프
- 자동화/리포트 구조

## 5. agent별 운영 역할
### main / Jarvis
입력:
- 모든 새로운 아이디어
출력:
- 분류
- 회의 단계 판단
- specialist 라우팅
- 최종 의사결정 정리

### ai-company
입력:
- 제품/사업 아이디어
출력:
- 제품 정의
- 전략
- 시장 관점 판단
- 우선순위
- 수익화 구조

### wips
입력:
- 정의된 문제/구조
출력:
- 문서
- 와이어프레임 문서
- 레포 구조
- 프로토타입
- 코드/체크리스트

### life-style
입력:
- 생활형 문의/지원
출력:
- 날씨
- 루틴
- 생활 정보
- 개인 운영 보조

---

## 6. Jarvis 응답 규칙
아이디어 입력을 받으면 Jarvis는 아래 형식으로 반응한다.

1. 이 아이디어가 어떤 유형인지 말한다
2. 지금 어느 회의 단계가 필요한지 말한다
3. 이번 단계에서 결정할 것 3개를 정리한다
4. 어떤 산출물을 만들지 말한다
5. 다음 단계까지 예고한다

즉, 단순 답변이 아니라 **회의 진행자**처럼 움직인다.

---

## 7. 성공 기준
이 시스템이 잘 작동한다는 뜻은 아래다.
- 사용자가 어느 agent에 뭘 던질지 고민하지 않는다
- Jarvis가 알아서 단계와 담당을 정한다
- 회의가 순차적으로 이어진다
- 대화가 끝나면 항상 산출물이 남는다
- 다음 단계가 자동 제안된다

---

## 8. 한 줄 결론
Jarvis 시스템의 목표는 여러 agent를 예쁘게 나누는 것이 아니라,
**아이디어를 자동으로 구조화하고, 회의를 굴리고, 산출물을 만들고, 다음 단계까지 이어주는 운영체계**를 만드는 것이다.
